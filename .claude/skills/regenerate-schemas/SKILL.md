# Regenerate the public JSON Schemas

`docs/schemas/analysis_config.json` and
`docs/schemas/analysis_results.json` are generated by
`scripts/generate_schemas.py` from the `__dataclass_fields__` of the
public dataclasses. They are **not** currently checked by CI or by
`pre-commit`, so drift goes silent — the schemas served at
`https://ranipdx-glitch.github.io/BVID-FE/schemas/` end up out of sync
with the code until a release notices.

When this skill is relevant: any edit that adds, removes, or retypes
a field on one of these dataclasses (or anything they transitively
reference):

- `bvidfe.analysis.config.AnalysisConfig`, `MeshParams`
- `bvidfe.analysis.results.AnalysisResults`, `FieldResults`
- `bvidfe.core.geometry.ImpactorGeometry`, `PanelGeometry`
- `bvidfe.core.material.OrthotropicMaterial`
- `bvidfe.damage.state.DamageState`, `DelaminationEllipse`
- `bvidfe.impact.mapping.ImpactEvent`

This also includes extending a `Literal` (e.g. `TierName`,
`LoadingMode`, `CriterionName`) — the schema emits these as `enum`s
and they must be refreshed when the union grows.

## Steps

### 1. Check whether regeneration would change anything

```bash
python scripts/generate_schemas.py --check
```

- Exit 0 with `schemas are up to date.` → nothing to do; stop here.
- Exit 1 with a list of files → the schemas are stale; continue.

### 2. Regenerate

```bash
python scripts/generate_schemas.py
```

This writes `docs/schemas/analysis_config.json` and
`docs/schemas/analysis_results.json` with deterministic encoding
(sorted keys, 2-space indent, trailing newline) so the diff is
reviewable.

### 3. Sanity-check the diff

```bash
git diff docs/schemas/
```

Look for:

- New / removed fields matching the source change (no stray rewrites).
- `enum` arrays under `tier`, `loading`, `criterion` containing the
  expected literals.
- No accidental ordering churn (the script sorts `$defs` and uses
  `sort_keys=True`, so a clean regen should diff only on real changes).

If the diff is much larger than the source change should have caused,
stop and investigate — most likely cause is a `typing.get_type_hints`
resolution issue from a forward reference that needs an explicit
import in `scripts/generate_schemas.py`.

### 4. Commit alongside the source change

The schema regen belongs in the **same commit** as the dataclass
edit that caused it. Don't split them — reviewers need to see the
schema delta next to the field delta to verify they match.

## Things this skill must never do

- Never hand-edit `docs/schemas/*.json` directly. The script's output
  is the source of truth.
- Never run `generate_schemas.py` with `--out-dir` pointing somewhere
  other than `docs/schemas/` unless the user explicitly asks for a
  preview elsewhere.
- Never skip the `--check` first if you're not sure whether a change
  is needed — `--check` is the cheapest way to know.
